# ADR-001: Python 3.11 stdlib-only runtime

## Status
Accepted

## Context
The taskq project is governed by a hard zero-runtime-dependency constraint originating in SPEC §1, restated in SRS §1, and reaffirmed in PROJECT_BRIEF §4. The product is a small local CLI whose value proposition is operational predictability, not feature breadth; supply-chain surface therefore dominates feature richness as a design concern. A single vulnerable transitive dependency — a hijacked `pip install` artefact, a typosquatted CLI helper, a yanked `requests`-class library pulled in for one convenience method — would invalidate the security posture of the entire tool and force a coordinated CVE disclosure across every user. SPEC §1 forbids this exposure outright, and the taskq codebase must therefore deliver a fully working CLI (subprocess orchestration, JSON persistence, atomic write, regex redaction, argparse dispatch, ISO-8601 timestamps, monotonic duration measurement) using only modules that ship with a CPython 3.11 interpreter. See SAD.md §1 (architecture overview) and SAD.md §6 (Technology Choices) for the canonical inventory of permitted modules.

The constraint interacts with several non-functional requirements. NFR-02 (security) requires shell-injection-safe subprocess invocation, which `subprocess.run` with `shell=False` provides natively. NFR-03 (reliability) requires atomic file replacement, which `os.replace` provides natively. NFR-01 (performance p95 < 50 ms for submit+status) is achievable because the hot path is `validate_command` + `atomic_write` + `load_store`, all of which are stdlib operations whose aggregate cost is well under budget on a warm filesystem cache (SRS §3 / SAD.md §5 `nfr_traceability.NFR-01`). The stdlib constraint therefore aligns with every measured NFR rather than competing with them.

## Decision
`taskq` ships as Python 3.11 standard library only at runtime. No third-party package may appear in `core/taskq/` imports, in any wheel/sdist `install_requires`, in any `pyproject.toml` `[project.dependencies]` block, or in any `requirements.txt` consumed at install time. The permitted import set is enumerated in SAD.md §6 and includes `subprocess`, `shlex`, `json`, `os.replace` (via `os`), `uuid`, `argparse`, `pathlib`, `re`, `datetime`, `time`, `dataclasses`, and `sys`. Test-only tooling (pytest, pytest-cov, hypothesis if needed) is provided by the development environment and is explicitly NOT a runtime dependency; it must never be imported from inside `core/taskq/`. A static-scan check (`test_redteam_shell_true_absent_in_codebase`, TEST_INVENTORY.yaml §25-test #21) enforces the related `shell=False` invariant and reinforces the stdlib-only posture by failing any import path that would re-introduce a shell surface.

## Rationale
The stdlib-only choice follows directly from SPEC §1, which the SAD inherits as a frozen requirement (no invention rule). Three forces make this choice the lowest-regret option. First, **portability**: a stdlib-only Python 3.11 program runs on any CPython 3.11+ install without a virtualenv, which matches the deployment model of a `python -m taskq` script invoked from cron, CI, or a developer's shell. Second, **auditability**: every line of behavior is traceable to a documented CPython module, eliminating the supply-chain risk that a transitive dep gets compromised and silently exfiltrates `tasks.json` or the secrets it is meant to redact. Third, **operational consistency**: stdlib APIs (`subprocess.run`, `os.replace`, `json`) have stable, well-understood semantics across versions, so the test suite does not need to track upstream breaking changes in a CLI framework.

Alternatives considered and rejected: (a) **Click / Typer** — rejected for two reasons: (i) it would violate the stdlib-only constraint, and (ii) `argparse` already covers the four subcommands (`submit`/`run`/`status`/`list`/`clear`) plus the global `--json` flag with no ergonomic loss. (b) **Rich / colorama** for human output formatting — rejected because the human-output surface is a single status line per subcommand (SAD.md §2.2.1), and adding a TUI library would inflate the dependency graph for no observable gain. (c) **Pydantic** for the `Task` dataclass — rejected because the Task record has four fields with primitive types (id, command, status, created_at), and a `@dataclass` with type hints provides equivalent validation and zero overhead. (d) **httpx / requests** — rejected because `taskq` is local-only and never makes network calls; no HTTP client is needed. (e) **pyyaml** — rejected because `tasks.json` is the only persistence format and `json` is sufficient. Each rejected alternative would have weakened either the security posture (supply chain) or the auditability (one more library whose CVE feed must be monitored).

## Consequences
- Positive: Zero supply-chain attack surface — a compromised PyPI package cannot affect `taskq` users. Trivial deployment (`python -m taskq`) with no virtualenv setup. Behavior is reproducible across machines because every imported module is bundled with the interpreter. The static-scan test `test_redteam_shell_true_absent_in_codebase` doubles as a stdlib-discipline check (any third-party import would be visible to the same scan). Documentation is one-to-one with the Python Language Reference, easing onboarding.
- Negative: Cannot use ergonomic third-party CLI helpers (no `click.echo`, no `typer.echo`, no `rich.console.print`); human output formatting must be hand-rolled in `cli/format.py::render_human`. No third-party validation library (must hand-write the injection-char blacklist in `store/validation.py`). No HTTP, no async, no batteries-included database — features that might tempt future scope creep are simply unavailable. The team must own every regex, every error code, every timestamp format.
- Neutral: Test tooling (pytest) remains outside the runtime boundary by convention; the `core/taskq/` → `tests/` import direction is one-way and is documented in SAD.md §2.1 (Design Principles).

# ADR-002: 4-module decomposition (cli / store / executor / config)

## Status
Accepted

## Context
`taskq` must be partitioned across one or more source modules, and the partition must satisfy three independent forces simultaneously. First, the **functional-requirement axis** of SRS §2 (FR-01 model+persistence, FR-02 subprocess+state machine+retry, FR-03 CLI dispatch) is a natural 3-way split that maps cleanly onto three responsibilities; a fourth, cross-cutting concern — environment-variable configuration — is also evident from SRS §6 (`TASKQ_HOME` / `TASKQ_TASK_TIMEOUT` / `TASKQ_RETRY_LIMIT`). Second, the **CRG community-cohesion scoring** described in SAD.md §2.1 penalizes a flat `src/` layout (zero internal edges → cohesion 0.0) and rewards 3-6 source directories, each with a hub module called from every sibling (Principles 1, 2, 3, 4, 6). Third, the **separation-of-concerns** principle in software architecture argues against a monolithic `taskq.py` that mixes `argparse` parsing, JSON I/O, subprocess spawning, and env-var resolution; the resulting tangle would be untestable in isolation and would couple the high-risk surface (subprocess + redaction) to the low-risk surface (formatting).

The forces point to the same answer only if the partition respects dependency direction: `cli` may import from `store`, `executor`, and `config`; `executor` may import from `store` and `config`; `store` may import from `config` only; `config` may import nothing from sibling modules. A circular `cli → store → executor → cli` loop would invalidate the entire architecture. The decomposition must therefore be acyclic (SAD.md §5 `architecture_constraints: no_circular_dependencies`) and must place the leaf module (`config`, no outgoing edges) at the bottom of the import DAG.

## Decision
`taskq` is decomposed into exactly four source modules, each occupying its own directory under `core/taskq/`: `taskq.cli/`, `taskq.store/`, `taskq.executor/`, `taskq.config/`. The module-to-responsibility map is: `cli` — argparse dispatch + format + exit-code mapping (FR-03); `store` — JSON persistence + validation + atomic write + corruption detection (FR-01); `executor` — `subprocess.run` wrapper + state machine + retry + redaction (FR-02 + NFR-03 redaction surface); `config` — env-var reader with defaults + per-process snapshot (SRS §6). Allowed dependency edges are: `cli → {store, executor, config}`, `executor → {store, config}`, `store → {config}`, `config → {}`. The split is published in the SAB block (SAD.md §5) under `layers:` and `allowed_dependencies:`; the `fr_module_traceability:` block pins FR-01 → `taskq.store`, FR-02 → `taskq.executor`, FR-03 → `taskq.cli`. Each directory contains a hub module (`cli/format.py`, `store/persistence.py`, `executor/runner.py`, `config/env.py`) with ≥2 hub functions called from every sibling per SAD.md §2.1 Principle 4.

## Rationale
The 4-module split is the only partition that simultaneously (i) aligns one-to-one with the FR axes (FR-01/02/03 → store/executor/cli) plus the cross-cutting config concern, (ii) keeps CRG community cohesion above the 0.3 floor by giving each directory a hub-and-spoke shape with explicit internal edges, and (iii) preserves a clean acyclic dependency DAG with `config` as the sink. Coarser partitions (a 2-module `cli+core` split) would collapse `executor` and `store` into one module, mixing the high-risk subprocess surface with the persistence invariants and dragging CRG cohesion down by inflating external edges in the entry point. Finer partitions (a 6+ module split with separate `validation`, `redaction`, `retry` modules) would push the directory count above the 3-6 sweet spot, dilute each community's internal-edge budget, and add import indirection that does not pay for itself at this codebase size (target: 4 source dirs, ≤50 nodes each per CRG Principle 6).

Alternatives considered and rejected: (a) **Flat single-module `taskq.py`** — rejected because it would fail CRG Principle 1 (single oversized community, cohesion 0.0) and would entangle subprocess lifecycle with JSON I/O, making the high-risk surface untestable in isolation. (b) **Two-module split (entry + core)** — rejected because it merges `store` and `executor` into one module, hiding the boundary between persistence (corruption detection, atomic write) and subprocess orchestration (state machine, redaction); SAD.md §1 names these as the two highest-risk surfaces and they must be separable for code review. (c) **Six-module split with per-feature modules** — rejected because the codebase is too small to populate six hubs above the 0.3 cohesion floor; the extra indirection would produce communities with sparse internal edges, failing CRG scoring. (d) **Pushing config into cli** — rejected because `store` and `executor` both consume the `Config` snapshot, and a circular `cli → store → cli` loop would be required if config lived in cli; the dependency direction `config → {}` is the only acyclic placement.

## Consequences
- **Positive**: One-to-one FR-to-module mapping makes code review trivial (reviewer can find FR-01 logic by `grep -r FR-01 core/taskq/store/`). CRG cohesion is kept above 0.3 because each module is its own community with a hub and ≥2 hub functions (SAD.md §2.1 Principles 1, 2, 4). Test attribution in SAD.md §7 (25-test inventory) maps cleanly to a single primary module per test, with secondary modules making the cross-module edges explicit. The acyclic DAG is enforced by the SAB `allowed_dependencies` block and by inspection in code review. The store/executor boundary isolates the two highest-risk modules (SAD.md §1) from each other and from the entry point.
- **Negative**: Four directories each with 2-5 files mean a larger import surface to maintain than a flat `taskq.py` would have; a developer adding a new feature must decide which module owns it, and the wrong choice (e.g. putting redaction logic in `cli`) is easy to make. Cross-module calls add one import line per dependency, slightly more boilerplate than a flat module. The hub-and-spoke pattern is unusual enough that contributors must read SAD.md §2.1 to understand why the hub file exists; onboarding cost is non-zero.
- **Neutral**: The `config` module is a leaf (no outgoing edges to other `taskq.*` modules), so it is the natural target for the unit tests that need to mock environment state (`test_config_env_keys_declared_in_env_example`); this is a side-effect of the DAG structure rather than a deliberate testability optimization.

# ADR-003: subprocess.run with shell=False + shlex.split

## Status
Accepted

## Context
FR-02 requires `taskq run <id>` to execute a previously submitted command as a subprocess, capturing stdout, stderr, and exit code, with a wall-clock timeout. NFR-02 (security) requires that the entire codebase remain free of `shell=True` so that user-supplied command strings cannot trigger shell metacharacter expansion (pipes, globs, command substitution, redirections, backgrounding). These two requirements sit in tension with the operational need to accept a *single command string* as user input (CLI ergonomics dictate `taskq submit "echo hi"`, not `taskq submit echo hi` with quoted argv), and the only way to honor both is to split the string into argv with `shlex.split` and hand the resulting list to `subprocess.run` with `shell=False`. Any other construction — `subprocess.run(command, shell=True)`, `os.system(command)`, `os.popen(command)`, `subprocess.Popen(command, shell=True)` — would either reintroduce a shell surface (defeating NFR-02) or require the user to type JSON-encoded argv at the CLI (defeating FR-03 ergonomics).

The risk is concrete and well-documented. A `shell=True` path is the canonical CWE-78 (OS Command Injection) sink: any unsanitized user input that contains `;`, `|`, `&`, `$()`, backticks, or redirection metacharacters is interpreted by `/bin/sh`, not by `execve`. The defense-in-depth pair — input blacklist at submit time (FR-01 `;` `|` `&` `$` `>` `<` `` ` ``) **and** `shell=False` at run time (NFR-02) — ensures that even if the validation layer is bypassed (e.g. a future code change accidentally inverts the blacklist), the executor cannot interpret the string as shell syntax.

## Decision
`taskq.executor.runner.run_subprocess` is the single, hard-coded entry point for subprocess invocation. It accepts a `command: str` and a `timeout: float`, performs `shlex.split(command)` to produce an argv list, and calls `subprocess.run(argv, shell=False, capture_output=True, text=True, timeout=timeout)`. `shell=False` is a hard invariant: any code path in `core/taskq/` that constructs a `subprocess` call with `shell=True` violates NFR-02 and is enforced by the static-scan test `test_redteam_shell_true_absent_in_codebase` (TEST_INVENTORY.yaml §25-test #21, SAD.md §7 row 21). The blacklist at submit time (six characters per FR-01) is a separate defense-in-depth layer; the two layers are independent and both must hold. `capture_output=True` is required to populate `stdout` and `stderr` on the `CompletedProcess` return value (FR-02 result fields `stdout_tail` / `stderr_tail`); `text=True` decodes bytes to `str` so that redaction can operate on lines. The `timeout` argument is required (no default) and is sourced from the per-process `Config.task_timeout` snapshot.

## Rationale
`subprocess.run` with `shell=False` is the only stdlib call that (i) bypasses the shell entirely (the kernel `execve`s the target binary directly, no `/bin/sh -c` indirection), (ii) accepts an argv list (the safe representation of a command), and (iii) supports the `timeout` + `capture_output` + `text` triple FR-02 requires in a single call. The `shlex.split` precondition converts the user-supplied string into the argv list that `shell=False` demands; without it, `shell=False` would receive a single-element list containing the literal command string, and the kernel would attempt to `execve` a file named `echo hi` (with a space) — the common misconception that `shell=False` is "safe by default" only holds when the caller already has an argv list. The combination `shlex.split` (caller side) + `subprocess.run(..., shell=False)` (executor side) is the canonical Python recipe for safe subprocess invocation and is the only one that satisfies NFR-02's "no shell surface" requirement.

Alternatives considered and rejected: (a) **`shell=True` with stricter input sanitization** — rejected because NFR-02 explicitly forbids `shell=True` anywhere in the codebase, and the static-scan test makes this contract machine-checkable. A single missed metacharacter (e.g. `\n` for newline injection) would defeat sanitization; `shell=False` removes the entire class of attacks. (b) **`subprocess.Popen` with manual `wait()` and `kill()` for timeout** — rejected because `subprocess.run(..., timeout=...)` already implements the timeout-then-kill pattern correctly (it raises `TimeoutExpired` after killing the child), and reimplementing it would re-introduce bugs the stdlib has already fixed. (c) **`os.system`** — rejected for the same reason as `shell=True`: it always invokes `/bin/sh -c`. (d) **`subprocess.run([command], shell=False)`** without `shlex.split` — rejected because the user supplies a single string, not a pre-split list; passing the string as a single-element argv would attempt to `execve` a binary whose name contains a space. (e) **Allowing shell features (pipes, globs) in the user command** — rejected because the FR-01 blacklist forbids the metacharacters, so the only shell features a user could legally request (e.g. `*` glob, `~` expansion) are also off-limits; this is the explicit NFR-02 trade-off.

## Consequences
- **Positive**: The CWE-78 attack surface is structurally eliminated — there is no `/bin/sh` in the call path, so metacharacter interpretation is impossible regardless of what the user submits. The static-scan test `test_redteam_shell_true_absent_in_codebase` makes the invariant a CI gate, so a future regression that re-introduces `shell=True` is caught before merge. `subprocess.run` is a single function call, so the subprocess lifecycle (fork/exec/wait/reap) is handled by the stdlib and not reimplemented. `TimeoutExpired` is raised by the stdlib itself when the timeout elapses, so the executor's timeout-handling code is reduced to a single `except` clause.
- **Negative**: Shell features (pipes `|`, command substitution `$()`, globs `*`, redirections `>`/`<`, backgrounding `&`, semicolons `;`) are unavailable to the user — but the FR-01 validation layer (SAD.md §2.2.2) rejects every metacharacter that would enable them, so this is the explicit NFR-02 trade-off rather than a usability gap. The user must type commands as single-token invocations (e.g. `taskq submit "ls /tmp"` works; `taskq submit "ls /tmp | grep foo"` does not). The `shlex.split` step adds one string-parsing pass per run, but the cost is negligible (microseconds on a string of length ≤ 1000 per FR-01).
- **Neutral**: The defense-in-depth pair (blacklist + `shell=False`) is duplicated logic in two modules (`store/validation.py` and `executor/runner.py`); this is intentional — if either layer fails, the other still holds.

# ADR-004: Atomic file write via tmp + os.replace

## Status
Accepted

## Context
`tasks.json` is the single point of truth for the task queue (SAD.md §1: "the store is the spine of the architecture"). It is read by every CLI invocation (`submit`, `run`, `status`, `list`, `clear`) and written by `submit` and `run`. NFR-03 (reliability) requires that the on-disk file always be a valid JSON document, even if the writing process is killed mid-write (risk R1: "concurrent / interrupted write corruption"). The natural failure modes are well-known: a power loss or `SIGKILL` between the `open(...).write(...)` and the corresponding `close()` leaves a truncated file, and the next `load_store()` raises `json.JSONDecodeError`. The `taskq` design must either prevent this class of failure or detect it and refuse to silently recover (FR-01's "no silent rebuild" rule).

The stdlib provides a canonical POSIX-atomic-write pattern: write the new contents to a sibling path (`tasks.json.tmp`), then `os.replace(tmp, final)`. `os.replace` is atomic on POSIX file systems when both paths reside on the same filesystem (the `rename(2)` syscall guarantees the directory entry swap is atomic). On Windows, `os.replace` is also atomic (it uses `MoveFileEx` with `MOVEFILE_REPLACE_EXISTING`). The pattern handles three failure modes correctly: (i) crash before `os.replace` → original file intact, `tasks.json.tmp` may be a partial garbage file (cleaned up on next successful write); (ii) crash after `os.replace` returns → new file is the new contents (the rename is atomic, so partial visibility is impossible); (iii) crash during `os.replace` → impossible (the kernel performs the swap atomically). The `test_reliability_kill_during_write_keeps_valid_json` and `test_reliability_concurrent_writes_do_not_corrupt` integration tests (SRS §8) verify the pattern holds under SIGKILL and concurrent writers.

## Decision
`taskq.store.persistence.atomic_write(path: Path, payload: str) -> None` is the single entry point for writing `tasks.json`. The function (i) writes `payload` to `tasks.json.tmp` in the same directory as `path` via a buffered `open(..., "w")` and `flush()` + `os.fsync()`; (ii) calls `os.replace(tasks.json.tmp, path)` to atomically swap the directory entry; (iii) is the only function in `core/taskq/` permitted to call `os.replace` and the only function permitted to write to the `tasks.json` path. The store module enforces this: `core/taskq/store/` is the sole importer of `json` and `os.replace` (SAD.md §2.2.2). The function is called from `submit_task`, `update_task`, and `clear_store` (the three write paths in the store). `load_store` calls `open(tasks.json).read()` then `json.loads(...)`; on `json.JSONDecodeError` the store raises `StoreCorrupted`, the CLI catches it at the top level, writes `store corrupted` to stderr, and exits 1 — **no silent rebuild** (FR-01 / NFR-03 anti-silencing rule, SAD.md §4).

## Rationale
`os.replace` is the only stdlib API that provides POSIX-atomic rename within a single filesystem, and the tmp-then-replace pattern is the canonical recipe referenced by Python docs, SQLite WAL documentation, and the Linux `rename(2)` man page. The same-FS requirement is satisfied trivially because the tmp file is created in the same directory as the target (`$TASKQ_HOME/`), which is by construction a single filesystem. The `fsync` step is a deliberate trade-off: it costs one disk-sync per write (sub-millisecond on SSD, single-digit milliseconds on spinning disk) but guarantees that the new file is durable before the rename, so a power loss between `fsync` and `os.replace` cannot resurrect stale data. Without `fsync`, the kernel may reorder the rename ahead of the data write, and a power loss could leave a zero-length file visible at the target path.

Alternatives considered and rejected: (a) **In-place rewrite with `open(path, "w")`** — rejected because a crash mid-write leaves a truncated file at the target path, triggering the `StoreCorrupted` path on the next invocation; this is the exact failure mode NFR-03 forbids. (b) **`tempfile.NamedTemporaryFile` + `shutil.move`** — rejected because `shutil.move` falls back to copy-then-delete on cross-FS moves (which is not atomic) and because the explicit `tasks.json.tmp` filename makes the on-disk state easier to reason about during postmortem. (c) **Write-ahead log (WAL) like SQLite** — rejected because the task queue's read/write volume is too low to justify the implementation cost; a tmp file per write is sufficient. (d) **`fcntl` file locking for concurrent writers** — rejected because (i) `fcntl` is POSIX-only and would break Windows compatibility, and (ii) the atomic-rename pattern already makes concurrent writes safe (the last writer wins, no corruption). (e) **Auto-rebuild on corruption** — explicitly forbidden by FR-01's "no silent rebuild" rule; the corrupted file is left on disk for forensic recovery, and the process exits 1.

## Consequences
- **Positive**: The on-disk file is always a valid JSON document (or the file is missing / was never written), satisfying NFR-03's "kill-during-write keeps valid JSON" requirement. The `test_reliability_kill_during_write_keeps_valid_json` integration test (SRS §8) passes by construction — a kill between `open(tmp).write()` and `os.replace()` leaves the original `tasks.json` untouched, and a kill after `os.replace()` leaves the new contents fully written. The pattern is stdlib-only, so ADR-001's no-third-party-deps posture is preserved. The `StoreCorrupted` path (FR-01) is exercised only when the file is genuinely corrupt (truncated, partially overwritten by a non-taskq process, manually edited into invalid JSON), making it a reliable signal of out-of-band corruption rather than a routine code path. Concurrent writers are safe: two `submit_task` calls racing produce two atomic renames, and the last writer wins without partial visibility.
- **Negative**: Each write costs two syscalls (write tmp, rename) plus one `fsync`; the latency is dominated by the `fsync`, which is ~1ms on SSD and ~5-10ms on spinning disk. The NFR-01 p95 < 50ms target (SAD.md §5 `nfr_traceability.NFR-01`) leaves ample headroom, so the cost is acceptable. The same-FS requirement means `TASKQ_HOME` cannot span multiple filesystems (e.g. a symlink to a different mount); this is a deployment constraint rather than a code defect, and it is documented in the `config.paths.resolve_home` docstring. A crashed write leaves a `tasks.json.tmp` orphan on disk; it is overwritten on the next successful write and never affects correctness, but it may surprise users who inspect `$TASKQ_HOME` manually.
- **Neutral**: The atomicity guarantee is "the directory entry swap is atomic," not "the disk write is atomic"; a power loss between `fsync` and `os.replace` could in theory leave a stale or zero-byte file visible, but in practice the kernel's write-back cache flushes the data page before processing the rename, and the test suite's SIGKILL coverage confirms the pattern holds on real hardware.

# ADR-005: Per-task state machine pending → running → done | failed | timeout

## Status
Accepted

## Context
Each task in `tasks.json` carries a `status` field that the executor reads at the start of `run_task` and writes back at the end (and at intermediate points during retry). FR-02 mandates a deterministic 4-outcome classification: a task that exits 0 is `done`, a task that exits non-zero is `failed`, a task that exceeds `TASKQ_TASK_TIMEOUT` is `timeout`, and a task that has been submitted but not yet run is `pending`. The retry policy (FR-02: "失敗/timeout 自動重試至 `TASKQ_RETRY_LIMIT` 次") reads the per-attempt outcome and decides whether to schedule another attempt; this decision is correct only if the state machine rejects illegal transitions (e.g. `running → pending`, which would let a `run` invocation reset a task to "not yet run" mid-flight).

The state machine is also the substrate for the exit-code mapping (ADR-007): `cli.cmd_run` consults the final `status` to decide between `sys.exit(0)` (done), `sys.exit(2)` (validation/unknown), `sys.exit(4)` (timeout), and `sys.exit(1)` (internal). A fifth state — e.g. `cancelled` for user-initiated abort — would have to be added to both the state machine and the exit-code table simultaneously, and the retry policy would need a rule for "do not retry a cancelled task." The design must therefore be conservative: the smallest set of states that covers every observable outcome, and a strict transition table that the executor enforces.

## Decision
The state machine has exactly **four states**: `pending`, `running`, and three terminal states `done`, `failed`, `timeout`. The legal transitions are: `pending → running` (executor picks up the task), `running → done` (subprocess exit 0), `running → failed` (subprocess exit non-zero), `running → timeout` (`subprocess.TimeoutExpired`). The transitions `pending → done` and `pending → failed` and `pending → timeout` are illegal and rejected by `taskq.executor.state_machine.apply_transition` (SAD.md §2.2.3 "Logical Constraints"). The transition `running → pending` (regression) is also illegal. Retry is modeled as repeated `running → {failed, timeout} → running` cycles, capped at `1 + TASKQ_RETRY_LIMIT` total attempts; only the **final** attempt's terminal state is persisted, intermediate attempts are not written back to the store (SAD.md §3.3). The CLI maps `status='timeout'` to `sys.exit(4)` in single-task mode and to `sys.exit(0)` in any batched/future background mode; the other terminal states always map to `sys.exit(0)` because the run completed without an external error.

## Rationale
A 4-state machine is the minimum expressive power that captures the three observable subprocess outcomes (clean exit, error exit, timeout) plus the unstarted state. A 3-state machine (pending / running / done) would conflate `failed` and `timeout` into `done` with an error code, losing the distinction the caller needs to decide between "your command is broken" (failed) and "your command is too slow" (timeout) — the latter is precisely what `TASKQ_RETRY_LIMIT` is meant to paper over, and the former is what the user needs to fix. A 5+ state machine (adding `cancelled`, `retrying`, `paused`, etc.) would expand the surface for no current requirement, and the SPEC explicitly rules out cancellation and pause features (SRS §4 "Out of scope: daemon, remote execution, non-JSON persistence, circuit breaker, cache, concurrency"). The strict transition table — not a permissive default — is required because a permissive FSM would let a buggy executor write `status='pending'` after a successful run, leaving the store in an inconsistent state.

Alternatives considered and rejected: (a) **Permissive FSM (any-to-any transitions)** — rejected because the store would have no way to distinguish a legitimate `pending → done` shortcut (illegal) from a buggy executor that forgot to write `running` first; the strict table makes such bugs surface as `apply_transition` raising `InvalidTransition`. (b) **3-state machine (pending / running / done) with an `error_code` sub-field** — rejected because the `error_code` would have to encode the `failed` vs `timeout` distinction, and downstream consumers (CLI exit-code mapping, retry logic) would have to inspect the sub-field on every read; promoting the distinction to a first-class state makes it visible at the store level. (c) **Adding a `cancelled` state for user-initiated abort** — rejected because cancellation is out of scope (SRS §4) and adding a state would also require a CLI subcommand to trigger it, expanding FR-03's surface. (d) **Persist intermediate retry attempts to the store** — rejected because it would bloat `tasks.json` and require a history list per task; only the final outcome is contractually meaningful to the caller (the retry policy is a detail of the executor, not the data model).

## Consequences
- **Positive**: The 4-state machine is small enough to verify by inspection (the legal-transition table fits in one screen) and is enforced by `apply_transition`, so a regression that introduces an illegal transition is caught at runtime. The retry policy is straightforward: `while attempts < 1 + TASKQ_RETRY_LIMIT and status in {failed, timeout}: run another attempt`. The CLI exit-code mapping is a pure function of the final `status` field, with no special cases for "intermediate retry failures." The `test_fr02_run_retry_limit_respected` integration test (SRS §8) verifies the cap by counting subprocess invocations, and the strict FSM makes the count deterministic. The `test_fr02_run_timeout_yields_timeout_and_exit_four` test asserts the timeout → exit 4 mapping directly.
- **Negative**: Any new outcome (e.g. a future `cancelled` state, or a `quota_exceeded` classification) requires extending the FSM and the CLI exit-code table simultaneously; the strict table means there is no "default" transition to fall back on. The `apply_transition` function must be called for every state change, which is one more call site to maintain than a permissive FSM. Persisting only the final attempt means the store does not record how many retries occurred; this information is logged to stderr by the executor (per `format.render_error`) but is not queryable via `taskq status <id>`.
- **Neutral**: The 4-state vocabulary is documented in SAD.md §2.2.3 and in the `Task` dataclass's `status: Literal["pending", "running", "done", "failed", "timeout"]` type hint, so the contract is visible to type checkers and to anyone reading the model definition.

# ADR-006: Secret redaction at executor boundary (sk-/token= → [REDACTED])

## Status
Accepted

## Context
NFR-03 (reliability) requires that the on-disk `tasks.json` never contain a secret, and the threat model in SRS §3 (risk R3: "secret 落盤洩漏") is concrete: a user submits a command like `curl -H "Authorization: Bearer sk-live-abc123..." https://api.example.com`, the executor captures `stdout_tail` and `stderr_tail` (each capped at 2000 chars per FR-02), and if the API responds with a debug payload that echoes the auth header (or if the command itself prints the env var to stderr), the secret lands in the persisted record. The next `taskq status <id>` would then echo the secret to the user's terminal, and the file would be backed up to any number of places (dotfiles repos, CI caches, crash reports) where the user did not intend to expose it. The risk is that the persistence layer cannot tell which lines are sensitive and which are benign, so the only safe default is to scrub every line that matches a known secret pattern *before* the write.

The pattern is encoded in the regex `sk-[A-Za-z0-9_-]{8,}` (catches `sk-` prefixed API keys, including Stripe-style `sk-live-...` and `sk-test-...` and OpenAI-style `sk-...`) and `token=\S+` (catches `token=` query-string parameters and HTTP headers). The choice of patterns is deliberately narrow: the goal is to catch the common leaks without false-positive noise that would make the store unreadable. The redaction is line-oriented: any line whose content matches either pattern is replaced in its entirety with the literal string `[REDACTED]`, so the surrounding context (timestamps, log level, function name) is also lost — this is the explicit trade-off documented in NFR-03.

## Decision
`taskq.executor.redaction.redact(text: str) -> str` is the single entry point for secret scrubbing. It applies the regex `(sk-[A-Za-z0-9_-]{8,}|token=\S+)` line-by-line and replaces any matching line with `[REDACTED]`. The function is called from `executor.runner.run_subprocess` (or from the post-processing step in `run_task`) on `stdout_tail` and `stderr_tail` *before* the `RunResult` is passed to `store.update_task`. The persistence contract is: **`store.update_task` MUST receive already-redacted tails**; the store does not re-redact (single responsibility). The redaction step is mandatory in the executor's hot path — there is no flag to disable it, and a `RunResult` that contains an unredacted secret is a bug. The redaction tests `test_redteam_secret_in_stdout_redacted_before_persist` and `test_redteam_secret_in_stderr_redacted_before_persist` (SRS §8, SAD.md §7 rows 19-20) verify the round trip: inject a known secret, run the task, read `tasks.json`, assert the secret string is not present.

## Rationale
The redaction must happen at the **executor boundary** (between subprocess completion and persistence) rather than at the persistence boundary (in `store.update_task`) or the read boundary (in `store.load_store` or `cli.cmd_status`) for three reasons. First, the executor is the only module that has access to the raw subprocess output before it is mutated by any other layer; placing redaction downstream (in the store or CLI) would require the secret to traverse the entire data path, multiplying the surface area for accidental leakage (e.g. a debug `print(stdout_tail)` in the store layer). Second, putting redaction in the store would couple the persistence module to a security policy, violating separation of concerns and making the store untestable without a secret-fixture harness. Third, putting redaction at read time would mean the on-disk file *contains* the secret — backups, crash dumps, and `git diff` history would all leak it, defeating the NFR-03 reliability goal. The boundary placement is therefore the only one that satisfies "secrets never on disk" rather than "secrets redacted on display."

Alternatives considered and rejected: (a) **Read-time redaction in `cli.format.render_human`** — rejected because the on-disk file would still contain the secret; the goal is "never on disk," not "never displayed." (b) **Persistence-time redaction in `store.update_task`** — rejected because it requires the secret-bearing text to traverse the executor-to-store call boundary, increasing the leakage surface; the executor is the trust boundary, not the store. (c) **Wider regex (e.g. any key=value with `password` or `secret` in the key name)** — rejected because the false-positive rate would make the stored records unreadable (every env-var dump line, every `?token=...` URL would be redacted). The narrow pattern is the explicit NFR-03 trade-off. (d) **No redaction, trust the user to scrub their commands** — rejected because the threat model is accidental leakage (debug echo, error message containing the env var), not malicious user input. (e) **External secret-scanning tool (e.g. `trufflehog`) at the CI layer** — rejected as a complement rather than a replacement, and not part of the runtime architecture; the in-process redaction is the only defense that runs on every write.

## Consequences
- **Positive**: Secrets are scrubbed before they reach the filesystem, satisfying the NFR-03 reliability goal. The redaction is enforced in the executor's hot path, so every write to `tasks.json` is automatically covered; there is no flag or config to disable it. The redaction test suite (`test_redteam_secret_in_stdout_redacted_before_persist`, `test_redteam_secret_in_stderr_redacted_before_persist`) provides regression coverage at the unit and integration levels. The narrow pattern keeps false positives to a minimum — a line containing `sk-` followed by 8+ word characters (very rare in normal output) or `token=...` (rare outside of HTTP query strings) is the only thing redacted. The redaction is line-oriented and idempotent, so re-running `redact` on already-redacted text is a no-op.
- **Negative**: **False positives are possible**: a benign log line that happens to contain `sk-` followed by 8 word characters (e.g. a checksum prefix, a stock ticker, a custom protocol marker) or `token=` (e.g. a config-file dump) will be replaced with `[REDACTED]`. This is the explicit NFR-03 trade-off — the alternative is leaking real secrets, which is strictly worse. A secret that does not match the two patterns (e.g. an OAuth refresh token, a PEM block, a JWT) will not be redacted; the redaction is a best-effort defense, not a complete one. The user is responsible for not printing secrets that the patterns do not catch, and the threat model assumes the two patterns cover the common cases. The redaction loses surrounding context (timestamps, log level, function name) on the matched line, which can make postmortem debugging harder when a redacted line would have been the smoking gun.
- **Neutral**: The redaction is in-process, not in-kernel; a kernel-level secret filter (e.g. `bpf` LSM hooks) would catch a wider class of leaks but is out of scope for a Python stdlib CLI. The two-pattern redaction is a pragmatic compromise documented in the `redact` function's docstring and in the NFR-03 row of the traceability matrix (SAD.md §5 `nfr_traceability.NFR-03`).

# ADR-007: Exit code mapping 0/2/4/1 with anti-silencing rule

## Status
Accepted

## Context
FR-03 mandates a uniform set of four exit codes for the entire CLI: `0` (success), `2` (validation error or unknown task id), `4` (task timeout in single-task mode), `1` (internal error). The mapping is the contract between `taskq` and the shell environment that invokes it (cron jobs, CI pipelines, `set -e` shell scripts, Makefile recipes); a caller that sees `2` knows the input was malformed, a caller that sees `4` knows the command ran but timed out, a caller that sees `1` knows something is structurally wrong with the tool itself. The mapping must be **uniform** (the same code means the same thing across all subcommands) and **complete** (every error path has a declared code; no path exits 0 silently after an error). The risk is that a future code change adds a new error path (e.g. a new validation rule, a new internal failure mode) and either exits 0 (silently swallowing the error) or exits 1 (overloading the "internal error" code so the caller cannot distinguish validation from corruption).

The second concern is the **anti-silencing rule**: a `try / except` block that catches `Exception` (or, worse, a bare `except:`) and does not re-raise or exit with a documented code is a silent failure mode. The corruption path (FR-01: "tasks.json 損壞 → exit 1 不靜默重建") is the canonical example: if a future change adds `try: ... except: pass` around the `load_store` call, the corruption is hidden from the user, the silent rebuild creates a new empty store, and the user loses their task history without warning. The same risk applies to validation errors (FR-01: "驗證規則違反 → exit 2, 不得寫入存儲") and to unexpected exceptions in the executor (FR-02: "其他未預期例外 → exit 1, 不得以裸 except: 吞噬").

## Decision
The CLI uses exactly four exit codes, declared as module-level constants in `taskq.cli.exitcodes`: `EXIT_OK=0`, `EXIT_VALIDATION=2`, `EXIT_TIMEOUT=4`, `EXIT_INTERNAL=1`. The mapping is enforced at the boundaries:
- `cli.cmd_submit` / `cli.cmd_status` catch `ValidationError` and `unknown task` and exit `EXIT_VALIDATION` (2).
- `cli.cmd_run` checks the final `RunResult.status`; if `timeout` and the call is in single-task mode, exit `EXIT_TIMEOUT` (4); otherwise exit `EXIT_OK` (0).
- `cli.main` (the top-level entry point) catches `StoreCorrupted` and exits `EXIT_INTERNAL` (1) with `store corrupted` written to stderr; it also catches any uncaught `Exception` and exits `EXIT_INTERNAL` (1) with the exception class name in stderr.
- No `try/except` block in `core/taskq/` may use a bare `except:` or a broad `except Exception:` to swallow an error without re-raising or mapping to a documented code (SAD.md §4 "Anti-silencing rule"). The rule is enforced by inspection in code review and by the `test_redteam_shell_true_absent_in_codebase` static scan (which is a related but distinct invariant).

## Rationale
A 4-code mapping is the minimum set that distinguishes the three externally-visible error categories (validation, timeout, internal) from success, and it is the standard convention for Unix CLI tools (cf. `grep`'s 0/1/2, `diff`'s 0/1/2, `pyflakes`'s 0/1). A 2-code mapping (0/1) would conflate validation errors with internal errors and force the caller to parse stderr to distinguish them, breaking `set -e` automation. A 5+ code mapping would add codes that no current caller needs (e.g. a separate code for "task not found" vs "validation failed") and would require the caller to maintain a lookup table; the SRS §2 "unknown task id" rule bundles both into code 2 deliberately. The anti-silencing rule is enforced by code review because there is no static analyzer that can reliably distinguish a legitimate `except SomeError:` (e.g. catching `subprocess.TimeoutExpired` to classify it as `status='timeout'`) from a swallowing `except Exception: pass`. The convention "no bare except, no broad except without re-raise or documented exit code" is the smallest rule that prevents the most common silent-failure pattern.

Alternatives considered and rejected: (a) **2-code mapping (0/1)** — rejected because the caller cannot distinguish "user typed a bad command" (validation, recoverable) from "the tool is broken" (internal, not recoverable) without parsing stderr; this defeats the purpose of exit codes. (b) **5+ code mapping (separate codes for each error class)** — rejected because the SRS §2 explicitly defines the 4-code contract and no caller has asked for finer granularity; the 25-test inventory (SRS §8) is fully covered by the 4 codes. (c) **Permitting bare `except:` for "defensive" cleanup** — rejected because the anti-silencing rule is binding (SAD.md §4) and a bare except is the canonical silent-failure pattern. (d) **Catching `Exception` at the top level and exiting 0 for "expected" failures** — rejected because it violates the uniform mapping (success must mean the operation completed; partial completion is not success). (e) **Using `sys.exit(0)` for validation errors with a stderr message** — rejected because `set -e` callers would not detect the failure; the 4-code contract exists precisely so shell scripts can branch on the exit code.

## Consequences
- **Positive**: The 4-code contract is uniform across all five subcommands (`submit` / `run` / `status` / `list` / `clear`), so a caller that scripts one subcommand can apply the same `case` statement to the others. The mapping is declared in a single module (`cli/exitcodes.py`) and imported by every handler, so a code change is a one-line edit. The anti-silencing rule prevents the most common silent-failure pattern (bare except, broad except without re-raise) and is enforceable by code review without a custom linter. The corruption path is loud: `StoreCorrupted` always exits 1 with `store corrupted` on stderr, so a corrupted `tasks.json` is immediately visible to the user. The `test_fr01_store_corruption_returns_one` integration test (SRS §8) locks in the corruption → exit 1 mapping.
- **Negative**: Every error path must declare its exit code, which is one more line of code per handler than a "log and continue" style would require. The anti-silencing rule forbids some legitimate-looking patterns (e.g. `except KeyError: return None`), forcing the developer to either re-raise, map to an exit code, or use a narrower exception class. The 4-code mapping does not distinguish "task not found" from "command invalid" (both are 2); callers that want to differentiate must parse stderr. The `test_redteam_shell_true_absent_in_codebase` static scan does not enforce the anti-silencing rule (it only checks for `shell=True`), so the rule relies on code review — a future contributor who adds `except Exception: pass` would not be caught by CI.
- **Neutral**: The mapping is documented in SAD.md §4 as a table (exit code → cause → handler location → user message → recovery) and in the `cli/exitcodes.py` module docstring, so the contract is discoverable from both the architecture document and the code.

# ADR-008: Test naming authority from P1 TEST_INVENTORY.yaml (no new FRs allowed)

## Status
Accepted

## Context
The test inventory that the project commits to (SRS §8, expanded in TEST_INVENTORY.yaml v1.1 to 45 function-level names) is declared in **Phase 1 (Requirements)** as the authoritative list of test functions. The D4 spec-coverage check in the harness (`harness/check_constitution.py` or its successor) unifies the P1 inventory, the P2 SAD cross-reference (SAD.md §7), and the P3 actual `def test_*` definitions in `tests/`, and reports drift if any of the three lists disagree. Adding a test function whose name is not in the P1 inventory would create a fourth list (P3 actual but not P1 declared) and would fail the unification check, blocking CI. Removing a test function from the P1 inventory would require a P1 amendment (re-approval of SRS.md), which is out of scope for Phase 2. The constraint is therefore: the set of `def test_*` names in `tests/` at the end of Phase 3 must be exactly the P1 inventory (plus parametrized expansions of existing names), and the SAD's cross-reference table (SAD.md §7) must point at the same names.

The constraint interacts with two design pressures. First, parametrized tests (`pytest.mark.parametrize`) let a single `def test_*` cover multiple input cases (e.g. the 6 injection characters in `test_fr01_submit_injection_chars_returns_two`); this is the preferred way to expand coverage without adding new test names. Second, helper functions (e.g. `make_task`, `assert_redacted`) may live in `tests/conftest.py` or `tests/_helpers/` and are not subject to the inventory check; the inventory only constrains `def test_*` functions that pytest discovers at the module level of a test file.

## Decision
The P1 test inventory (SRS.md §8, 25 tests; TEST_INVENTORY.yaml v1.1, 45 function-level names) is the **naming authority** for Phase 3. The SAD's cross-reference table (SAD.md §7) must list the same names; P3 `tests/` may only contain `def test_*` functions whose names appear in the inventory, plus parametrized expansions of those names. New test cases that require new function names are **not allowed** in Phase 2 or Phase 3; the only way to add coverage is to (i) parametrize an existing test with additional inputs, (ii) add a helper function to `conftest.py` that is called from an existing test, or (iii) amend the P1 inventory via a SRS revision (out of scope for this phase). The 25-test cross-reference in SAD.md §7 is the binding Phase 2 ↔ Phase 1 reconciliation; the 45-test expansion in TEST_INVENTORY.yaml is the binding Phase 1 ↔ Phase 3 reconciliation that `harness/build_trace.py` enforces at P3.

## Rationale
The P1 inventory is the only place where the test names are declared with the FR/NFR they cover; allowing new names to be invented in P2 or P3 would break the traceability from test → FR → module, which is the whole point of the spec-coverage check. The harness D4 unification is what makes the coverage check trustworthy: if a test name is in P3 but not in P1, the harness cannot tell which FR it covers, and the coverage report undercounts (or overcounts) the FR/NFR roll-up. The parametrization escape hatch (one test name, many inputs) lets coverage grow without expanding the name set; the 6-character injection test (`test_fr01_submit_injection_chars_returns_two`, parametrized over `;` `|` `&` `$` `>` `<` `` ` ``) is the canonical example, and the parametrized list must match the FR-01 blacklist exactly (SAD.md §2.2.2). The 25-vs-45 reconciliation in SAD.md §7 is explicit: the 25 are the §8 baseline, the 45 are the function-level decomposition used by `build_trace.py`, and the architecture is unchanged by the difference (every 45-test is absorbed by one of the four primary modules).

Alternatives considered and rejected: (a) **Allowing P3 to add new test names freely** — rejected because the D4 unification would flag every new name as drift and CI would block; the harness cannot tell which FR a free-form new name covers. (b) **Pinning the P1 inventory to the 25-test SRS §8 list only** — rejected because the 45-test expansion in TEST_INVENTORY.yaml is the function-level decomposition that the actual `def test_*` definitions must match; pinning to 25 would under-specify the test surface and force `build_trace.py` to invent names. (c) **Defining a "test spec" document in P2 that supersedes the P1 inventory** — rejected because P2 is downstream of P1; the inventory is declared at P1 and P2 may only reference it, not redefine it. (d) **Auto-generating test names from a template (e.g. `test_fr01_<rule>_<outcome>`)** — rejected because the inventory names are hand-curated to be readable in CI output; auto-generation would produce names like `test_fr01_validation_injection_chars_returns_two` that are correct but harder to scan in a failure log.

## Consequences
- **Positive**: The spec-coverage check is a single source of truth: P1 declares, P2 cross-references, P3 implements, and the harness D4 unification verifies all three agree. New test cases that follow the parametrization pattern (e.g. adding a 7th injection character) require only a one-line edit to the parametrize decorator, not a new function. The traceability from test → FR → module is preserved end-to-end (e.g. `test_fr02_run_timeout_yields_timeout_and_exit_four` → FR-02 → `taskq.executor` → `taskq.cli`). The 45-test expansion in TEST_INVENTORY.yaml is the contract for `build_trace.py`, so the Phase 3 reconciliation is mechanical and does not require re-reading the SAD. The CRG `fr_module_traceability:` block in SAD.md §5 pins FR → module, and the inventory pins test → FR, so test → module is a transitive closure.
- **Negative**: A genuine new test case (e.g. a regression test for a future bug) that does not fit any existing inventory name cannot be added without a P1 amendment; the workaround is to parametrize the closest existing test, but this can stretch the test's intent (e.g. a timeout test that also exercises a corrupted-store recovery is awkward to express as a parametrization of `test_fr02_run_timeout_yields_timeout_and_exit_four`). The 25-vs-45 reconciliation is a documentation burden — the SAD must explain why two inventories exist and which is the binding contract for which phase. The constraint that `def test_*` names must be in the P1 inventory means a developer who writes a test first (TDD) and then tries to register it must either rename it to match the inventory or file a P1 amendment; this is friction that TDD-friendly projects do not normally have.
- **Neutral**: The inventory is enforced by the harness, not by the test runner; a developer who adds an undeclared test function locally will see it pass under pytest but will fail the D4 unification check in CI. The constraint is therefore visible only at the CI boundary, not at the test-writing boundary.
